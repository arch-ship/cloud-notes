Introduction to Cloud Computing

Cloud computing is the delivery of computing services—servers, storage, databases, networking, software, analytics—over the internet to offer faster innovation, flexible resources, and economies of scale.

NIST's 5 Essential Characteristics of Cloud Computing

★ Exam Tip

Cloud Service Models

Cloud Service Models — darker = provider managed, lighter = user managed

Cloud Deployment Models

🏢

Public Cloud

AWS, Azure, GCP. Shared infrastructure over internet. Cost-effective, scalable.

🔒

Private Cloud

Dedicated to one org. On-premises or hosted. Full control, high security.

🔀

Hybrid Cloud

Mix of public + private. Sensitive data private, scalable workloads public.

☁️

Multiple public providers simultaneously. Avoids vendor lock-in.

Type 1 vs Type 2 Hypervisors

Type 1 — Bare Metal

  • Runs directly on hardware (no host OS)
  • Better performance, lower overhead
  • Strong isolation, smaller attack surface
  • Used in enterprise data centers
  • Examples: VMware ESXi, Hyper-V, Xen

Type 2 — Hosted

  • Runs on top of existing host OS
  • Easy to install alongside apps
  • Lower performance (extra OS layer)
  • Used for developer laptops, testing
  • Examples: VirtualBox, VMware Workstation
★ T1 Q3
Enterprise secure data center → Type 1. Developer laptop for testing → Type 2.

Microservices Architecture

An architectural style structuring an app as a collection of small, independently deployable services, each running its own process and communicating via lightweight mechanisms (REST APIs, message brokers).

Monolithic vs Microservices — independent services with isolated databases

★ T1 Q4
Company wants to scale only Cart service during seasonal sales → Microservices. Monolithic would require scaling the entire app wastefully.

Key Characteristics

  • Independently deployable and loosely coupled
  • Organized around business capabilities (Cart, Inventory, Payment)
  • Each service owns its own tech stack and database
  • Communicate via REST APIs, event streaming, or message brokers
  • Stateless processes — no session stored locally; state in external stores
  • Concurrency — multiple instances run in parallel

Node.js, Express.js & Microservices Code

Node.js is an open-source server environment using JavaScript on the server. Non-blocking, asynchronous I/O via a single-threaded event loop. Express.js is its minimal web framework for building REST APIs.

Microservices Pattern (★ T2 Q1 — 10 marks)

★ Exam Pattern
T2 Q1 = full microservices app. Make-up Q2 = blogging REST API. Always: setup → array store → 4 CRUD routes → listen on port.
// product-service.js — Port 3001
const express = require('express');
const app = express();
app.use(express.json());
let products = []; let id = 1;

app.get('/products', (req, res) => res.json(products));
app.post('/products', (req, res) => {
  const p = { id: id++, ...req.body }; products.push(p); res.json(p);
});
app.put('/products/:id', (req, res) => {
  const p = products.find(p => p.id == req.params.id);
  if (p) { Object.assign(p, req.body); res.json(p); }
  else res.status(404).send('Not found');
});
app.delete('/products/:id', (req, res) => {
  products = products.filter(p => p.id != req.params.id);
  res.send('Deleted');
});
app.listen(3001);

// api-gateway.js — Port 3000
const { createProxyMiddleware } = require('http-proxy-middleware');
app.use('/products', createProxyMiddleware({ target: 'http://localhost:3001', changeOrigin: true }));
app.use('/orders',   createProxyMiddleware({ target: 'http://localhost:3002', changeOrigin: true }));
app.listen(3000);

Docker & Containerization

Docker is an open platform for developing, shipping, and running applications in containers — isolated, portable environments. Solves "works on my machine" problem.

Docker Architecture: Client → Daemon (manages images & containers) ↔ Registry

Dockerfile — Every Instruction Explained

WORKDIR /app             # Set working directory inside container

COPY . .                 # Copy rest of source code
EXPOSE 3000              # Document the port app uses
CMD ["node", "index.js"] # Command when container STARTS

# Build and run:
docker build -t myapp:v1 .
docker run -it -p 3000:3000 myapp:v1

Docker Swarm — Container Orchestration

Docker Swarm is Docker's native clustering tool. It groups Docker hosts into a cluster (swarm) and automates deployment, scaling, and management of containers across nodes.

Docker Swarm: Manager dispatches tasks to workers. Self-healing replaces crashed containers.

Swarm CLI Commands

docker swarm init
Initialize swarm (current node = manager)
docker swarm join --token <TOKEN> <IP>:2377
Worker joins swarm
docker swarm join-token worker
Get worker join command/token
docker swarm leave
Node leaves swarm (--force for manager)
docker node ls
List all nodes in swarm
docker node promote <node>
Promote worker → manager
docker service create --name web --replicas 3 nginx
Create service with 3 replicas
docker service ls
List all services
docker service ps web
List tasks of a service
docker service scale web=5
Scale service to 5 replicas
docker service rm web
Remove service

Benefits of Orchestration

🤖 Automation

Auto deployment, scaling, management. Reduces human error, speeds release.

📈 Scalability

Add/remove containers based on demand. Handles traffic spikes automatically.

💓 High Availability

Containers distributed across nodes. Auto-failover on failure.

♻ Self-Healing

Detects crashed containers, auto-creates new task. Desired state maintained.

⚖ Load Balancing

Built-in service discovery. Routes traffic to healthy instances.

🔧 Resource Optimization

Efficiently allocates containers to available resources across nodes.

AWS ECS — Elastic Container Service

Amazon ECS is a highly scalable container orchestration service managing Docker containers on EC2 instances or AWS Fargate. Handles scheduling, scaling, and load balancing of containerized apps.

ECS Cluster → Service → Tasks (running containers pulled from ECR)

Key ECS Components

ComponentDescription
ECRElastic Container Registry — AWS's private Docker image registry. Push your image here before deploying.
ClusterLogical grouping of tasks/services. Can have multiple services (auth, db, api…)
Task DefinitionBlueprint: image, CPU/memory, ports, env vars. Like a Dockerfile for the whole task.
TaskRunning instance of a task definition. Created/managed by a Service.
ServiceMaintains desired count of tasks, restarts crashed ones, handles scaling and load balancing.

ECS Deployment Steps

Build Docker image: docker build -t myapp .

Push to ECR: tag image and push using ECR push commands from AWS Console

Create ECS Cluster — EC2 or Fargate launch type

Create Task Definition referencing ECR image URI, CPU/memory, port

Create Service specifying task, replica count, load balancer

Configure Auto Scaling — scale when CPU > 70%, min 1 / max 10

Access via Load Balancer DNS endpoint

EC2 vs Fargate Launch Types

EC2 Launch Type

  • Containers on EC2 instances you manage
  • You handle OS patching, capacity
  • More control, more responsibility

Fargate (Serverless)

  • AWS manages infrastructure completely
  • You define task (CPU/memory) only
  • Pay only for vCPU + memory used

PPT 12 — Kubernetes | Highest Priority | Master architecture well

10

Kubernetes (K8s)

Open-source container orchestration tool. Written in Go, developed by Google, donated to CNCF. Groups containers into logical units. Works with public, hybrid, on-premises cloud. K8s = 8 letters between K and s.

Kubernetes Master-Worker Architecture. Master controls; Workers run pods. Pod crashes → auto-recreated.

Master Node Components

ComponentRoleKey Detail
API ServerEntry point for ALL REST commandsUsers interact via kubectl. Validates and configures API objects.
SchedulerAssigns pods to worker nodesTracks resource utilization. Places pods where resources are free.
Controller ManagerEnsures current = desired stateNon-terminating loop. Runs replication, endpoint, namespace controllers.
etcdDistributed key-value storeStores entire cluster state, config maps, subnets. Written in Go.

Worker Node Components

ComponentRoleKey Detail
KubeletPrimary node agentGets pod specs from API server. Ensures containers are running and healthy. Tries to restart failed pods.
Kube-proxyCore networking componentMaintains network across nodes/pods/containers. Load balancer for TCP/UDP. Exposes services externally.
PodsSmallest deployable unitGroup of containers on same host. Ephemeral — crash = new pod with NEW IP. Use Services for stable IPs.
DockerContainer runtimePackages and runs containers inside worker nodes.
★ Remember
Pod crashes → Kubernetes auto-creates a new pod with a NEW IP. Old IP breaks. That's why Kubernetes Services exist — to provide a stable network endpoint regardless of which pods are running.
